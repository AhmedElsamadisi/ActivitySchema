A data modeling standard for modern warehouses

One activity stream. Every analysis.

Activity Schema replaces sprawling dependency graphs with a single, time-series model of what entities do. Model business events once, then assemble them into metrics, cohorts, reports, and analysis without rebuilding the warehouse every time the question changes.

Built to remove modeling complexity.

Traditional schemas create many definitions, many joins, and many layers. Activity Schema keeps the modeling layer flat and lets analysis happen from a consistent event structure.

One definition per concept

Each activity is defined once and reused anywhere it matters.

No fragile dependency graph

Downstream queries depend on the activity stream, not a maze of intermediate models.

Reusable analysis

Because every implementation has the same shape, queries can be adapted by swapping activity names.

The core table is intentionally small.

The activity stream captures the minimum shared structure needed to relate every business event over time.

Column Purpose
activity_idUnique identifier for the activity record.
tsTimestamp, in UTC, for when the activity occurred.
customerThe globally unique identifier for the modeled entity.
activityThe business action, such as completed_order or submitted_ticket.
feature_1..3Compact, activity-specific metadata.
revenue_impactRevenue or cost associated with the activity.
activity_occurrenceThe number of times this entity has done this activity so far.
activity_repeated_atThe timestamp of the next instance of the same activity for the same entity.

Queries relate activities by entity and time.

Instead of hunting for foreign keys, you join the stream to itself and ask how activities happened before, after, or between other activities.

SELECT
  DATE_TRUNC('month', ts) AS month,
  COUNT(1) AS total_orders,
  SUM(revenue_impact) AS total_revenue
FROM customer_stream
WHERE activity = 'completed_order'
GROUP BY month
ORDER BY month;
